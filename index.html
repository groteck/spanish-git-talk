Git control de versiones

Un poco de historia:
  • Inventado por Linus Torvalds.
  • Gran potencia colaborativa.
  • Perfecto para el trabajo en grupo.
  • Usado y mantenido por una gran comunidad.
  • Las redes de código mas grande del mundo lo usan Github, Bitbuket, google code ...
Conceptos básicos:
  • HEAD
  • .gitignore
  • origin
  • master
HEAD
Se conoce como HEAD al commit en el que se encuentra el repositorio git, pudiéndose usar como referencia para desplazarse como veremos mas adelante.
.gitignore
Para no incluir ficheros innecesarios a nuestro repositorio git, o ficheros que no queremos compartir (ficheros de configuración personales, ficheros temporales, ...), se añaden a un fichero llamado .gitignore.

Ejemplo de fichero .gitignore
*.c
/database/config.yml
...
origin
En muchas ocasiones, sobre todo al principio, los usuarios de git no entienden que significa cuando usan en un comando "origin", origin es una de las formas mas comunes de llamar a un repositorio remoto, se puede cambiar por cualquier otro nombre, pero es muy común llamar así al repositorio principal, siempre se suele usar origin y cuando se añade un repositorio remoto si no se especifica se añade por defecto a esta palabra.

En este caso se actualiza el repositorio local con el contenido de nuestra rama.
~$ git push origin master
master
La rama principal, como con origin también es modificable, pero se toma como estandar ese nombre y es el que git le asigna por defecto a la rama con la que se inicializa un repositorio.

La usamos en el ejemplo anterior, pero no comentamos nada sobre la rama, la rama sobre la que hacemos la actualización es master.
~$ git push origin master
Aprendiendo mediate ejemplos:
  • Crear un repositorio.
  • Añadir nuestros archivos al repo.
  • Guardar nuestro primer punto de "restauración".
  • Crear una nueva rama.
  • Movernos entre ramas.
  • Mergear ramas.
Crear un repositorio:
Para crear un repositorio, primero debemos entrar en la carpeta donde este el código que queremos añadir al repositorio, y luego en el, lanzar la orden:
~$ git init
Añadir nuestros archivos al repo:
En git realmente podemos decir que es una capa de abstracción sobre el proyecto, no tenemos que compartir necesariamente todo el código que tiene nuestro proyecto, en ocasiones como nombramos antes al explicar el .gitignore, puede haber archivos que no se compartan nunca y no entren en el repositorio git, pero ¿y si realmente lo que queremos es añadir los cambios de un fichero y no nos interesa el resto?, en este caso el fichero .gitignore no nos serviria, una razón para esto seria por ejemplo no querer añadir aun los cambios a un punto de restauración, dado que los quieras agrupar por funcionalidad por ejemplo.
Para añadir uno o varios ficheros al próximo punto de restauración es:
~$ git add fichero1 fichero2
si queremos añadir todo el directorio podemos usar:
~$ git add .
Guardar nuestro primer punto de "restauración":
La forma de guardado de git no es automática, dado que hay datos sensibles que no se deben guardar automáticamente, como ya hemos comentado anteriormente, por ello y para poder navegar fácilmente entre versiones, git nos da los commits, que en esencia son diferencias de código entre versiones, el guarda solo las partes relevantes para la siguiente versión la versión actual, esta "fabricada" con trozos de las versiones anteriores y las nuevas diferencias que ella añade.
Para crear un commit tenemos dos maneras:
~$ git commit -m "resumen del commit"
Con este te abre un editor en el que editar el mensaje en lugar de incluirlo con el "-m" como en el comando anterior:
~$ git commit
Crear una nueva rama(git branch):
Hay dos formas de hacerlo, primero la versión clásica, que es pasándole el nombre de la rama a crear.
Para crear una rama:
~$ git branch nombre-para-la-rama
Crear una nueva rama(git checkout):
Y la siguiente forma de hacerlo.
Para crear una rama y moverte a ella en el mismo paso:
~$ git checkout -b nombre-para-la-rama
Movernos entre ramas:
Movernos entre ramas es fácil, solo hay que tener en cuenta que para poder movernos, los cambios de la rama actual han de estar comiteados, es decir, no podemos tener cambios sin "guardar".
Para movernos entre ramas ya existentes:
~$ git checkout nombre-de-la-rama
Mergear ramas(1):
Para mergear, merge es la acción de mezclar los commits de dos ramas, es decir incrustar una rama sobre otra, esto es complejo a priori, pero la verdad que git lo solventa bastante bien, mezcla por fechas los commits y al final de todo si nota algún conflicto nos dice que ficheros mirar en esos ficheros nos dice que códigos, son los que entran en conflicto, el conflicto en un merge se suele dar porque dos usuarios cambian la misma linea y el es incapaz de decidir cual de los códigos ha de quedarse en la linea, o si debe ser uno que contenga la funcionalidad de las dos lineas.
este es el aspecto de un fichero git con un conflicto:
<<<<<<<<<<<<<<< HEAD
linea de la rama de git en la que te encuentras
=================
linea de la rama que mergeaste con la tuya
>>>>>>>>>>>>>>> id del commit que fallo
Mergear ramas(2):
Para mergerar ramas se usa el comando merge y se fusiona la rama en la que esta el usuarios con una rama externa, es decir, la rama que se queda con la mezcla, es la rama desde la que se lanza la orden, la otra rama no se ve modificada:
~$ git merge nombre-de-rama
